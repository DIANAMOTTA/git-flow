Git Flow

Branchig Model

Git Flow é um modelo de ramificação para organizar fluxo de trabalho de desenvolvimento de software.

O Git Flow define regras e comandos para criar e gerenciar branches com finalidades específicas, como:

  • master (ou main): versão de produção (código estável).
  • develop: versão de desenvolvimento (código em andamento).
  • feature/*: novas funcionalidades.
  • release/*: preparação para uma nova versão.
  • hotfix/*: correção urgente em produção.

    • Fluxo de Trabalho do Git Flow

  1. master —> produção
  2. develop —> base para features
  3. feature/ —> branch criada a partir de develop
  4. release/ —> criada a partir de develop, depois merge com master e develop
  5. hotfix/ —> criada a partir de master, depois merge com master e develop

Conventional Commits

Conventional Commits é um padrão que estabelece um formato claro e padronizado para mensagens de commit. Ele facilita tanto o entendimento humano quanto a automação, como geração de changelogs e versionamento semântico

Estrutura básica:

        
           <tipo> (escopo): <descrição>
           
           [corpo opcional]

           [rodapé(s) opcional(is)]
  • tipo: feat, fix, docs, style, refactor, test, chore, entre outros
  • escopo: área afetada entre parênteses (ex: (api), (auth))
  • descrição: resumo curto e direto
  • corpo: detalhes do “por quê” e decisões
  • rodapé(s): informações extras, como BREAKING CHANGE ou links para issues

Tipos de commits mais usados

  • feat: Novo recurso (define MINOR do SemVer)
  • fix: Correção de bug (define PATCH do SemVer)
  • docs: Documentação
  • style: Formatação sem afetar lógica
  • refactor: Refatoração sem correção ou novo recurso
  • test: Adição ou ajuste de testes
  • chore: Tarefas de manutenção (deps, CI…)
